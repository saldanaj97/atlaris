## Validation Schema + Type Split

Primary commit: `819351e0` `feat(billing, pdf, plans): introduce new validation schemas and refactor existing structures`

What shipped:

- Split `src/features/billing/validation/stripe.ts` into a `.schemas.ts` + `.types.ts` pair so Zod schemas and inferred types stop sharing one file with wrapper logic.
- Applied the same `.schemas.ts`/`.types.ts` split to PDF validation (`src/features/pdf/validation/pdf.ts`) and learning-plan validation (`src/features/plans/validation/learningPlans.ts`).
- Kept the existing `pdf.ts` / `learningPlans.ts` / `stripe.ts` entrypoints as thin re-export shims so consumers don't break.
- Pulled a set of incidental cleanups through the same commit: `src/lib/api/parse-json-body.ts`, `src/lib/api/pdf-rate-limit.ts`, and `src/lib/db/queries/plans.ts` saw small drift fixes against the new validation surfaces.

Why it mattered:

- The `pdf.ts` and `learningPlans.ts` validation files had grown large enough that schemas, types, and helper functions were being re-exported from the same module, which made it hard to see which imports were canonical.
- The `.schemas.ts` + `.types.ts` pattern mirrors the existing `src/shared/schemas/*.schemas.ts` layout and makes the canonical source for each shape explicit.

Notable file areas:

- `src/features/billing/validation/stripe.{ts,schemas.ts,types.ts}`
- `src/features/pdf/validation/pdf.{ts,schemas.ts,types.ts}`
- `src/features/plans/validation/learningPlans.{ts,schemas.ts,types.ts}`
