## Regeneration Quota Reservation Boundary

Primary commit: `54e4ea68` `refactor(billing): introduce regeneration quota reservation boundary`

What shipped:

- Replaced the route-local reserve/enqueue/decrement choreography in `src/app/api/v1/plans/[planId]/regenerate/route.ts` with a billing-owned `runRegenerationQuotaReserved()` boundary backed by a private metered-reservation core.
- Added `src/features/billing/metered-reservation.ts` so the core hands the caller a `MeteredReservationToken` capturing the exact month bucket reserved — compensation can never drift to a different month even if the request crosses midnight or a month boundary between reserve and rollback.
- Added `src/features/billing/regeneration-quota-boundary.ts` as the single public entrypoint; the boundary catches throws inside `work()`, compensates exactly once, and rethrows the original error with reconciliation telemetry kept fire-and-forget so a flaky Sentry call cannot shadow the caller's error.
- Left route-edge concerns (auth, ownership, early-active-job conflict, rate limiting, response headers, inline drain) untouched; HTTP semantics for the 202 success, 409 deduplicated, and 429 quota-denied paths stay behaviorally identical.
- Added `tests/integration/features/billing/regeneration-quota-boundary.spec.ts` covering the success, compensation, double-compensation-prevention, and month-boundary paths.

Why it mattered:

- Quota reserve/rollback was interleaved with job enqueueing and worker drain in the route, which made it hard to reason about what happens when the enqueue throws midway through.
- Moving the boundary into billing with an explicit reservation token gives regenerate and later PDF-lifecycle callers one tested place for the "reserve → work → compensate on failure" pattern.

Notable file areas:

- `src/features/billing/metered-reservation.ts`
- `src/features/billing/regeneration-quota-boundary.ts`
- `src/features/billing/quota.ts`
- `src/app/api/v1/plans/[planId]/regenerate/route.ts`
- `tests/integration/features/billing/regeneration-quota-boundary.spec.ts`

Scope note:

- This is phase 1 of issue #304. PDF lifecycle migration onto the same private core, removal of the unused `QuotaPort.reservePdfQuota()` / `rollbackPdfQuota()`, and shrinking the coupled lifecycle rollback tests are deferred to phase 2.
