## PDF Plan Creation and Extraction Removal

Primary commit: `8f86f0b2` `feat!: remove PDF plan creation and extraction pipeline`

What shipped:

- Deleted PDF upload and extraction UI, the `from-pdf/extract` route, the
  `src/features/pdf` module (validation, structure, security scanners), hooks,
  and PDF-specific lifecycle strategies and adapters.
- Added migration `0027_windy_agent_zero.sql` to drop PDF-era columns
  (`extracted_context`, usage metrics keyed to PDF generation), narrow
  `plan_origin`, and coerce legacy `origin='pdf'` rows to `manual`.
- Updated OpenAPI, shared schemas, env and config, billing usage types, AI
  prompts/constants, and the plan-creation and pricing surfaces for a
  manual/AI-only origin model.
- Refreshed architecture and testing docs; added `docs/development/deploy.md`
  cutover guidance; removed obsolete PDF e2e and security tests.
- Logged follow-up sequencing in `.daily-recap/2026-04-20/plans/pdf-removal-followups-plan.md`
  and `.plans/005-pdf-removal-followups/`; extended `.plans/lessons.md`.

Why it mattered:

- PDF was the highest-maintenance alternate origin: uploads, AV, extraction,
  and parallel types. Removing it collapses the product surface and the schema
  to the origins you intend to support going forward.
- The breaking change is paired with a single numbered migration and explicit
  deploy documentation so operators can order app rollout and DDL safely.

Notable file areas:

- `src/lib/db/migrations/0027_windy_agent_zero.sql`
- `src/features/plans/lifecycle/` (factory, creation pipeline, ports, types)
- `src/app/plans/new/` and removed PDF components

## Integration Truncate Hardening

Primary commit: `e796a937` `fix(test): harden integration truncate and reduce lock contention`

What shipped:

- Single multi-table `TRUNCATE` in a transaction with `lock_timeout` and retry
  on deadlock / lock-not-available; included FK-related tables in one
  statement for deterministic locking.
- Dropped redundant per-spec deletes and `resetDb` where file-level isolation
  already applied; aligned the unit DB mock transaction shape with
  `tx.execute`.

Why it mattered:

- Parallel integration workers were contending on table locks; batching
  truncates and retrying transient lock failures stabilizes the suite on the
  forked DB stack.

Notable file areas:

- `tests/helpers/db/truncate.ts`
- `tests/setup/db-provisioning.ts` (contributor notes in follow-up commit)
