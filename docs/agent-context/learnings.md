# Learnings

Persistent user preferences and durable workspace facts maintained by the `continual-learning` skill. **Whenever you read or follow `[AGENTS.md](../../AGENTS.md)`, read this file as well.**

## Learned User Preferences

- When the user indicates that a **subagent or explore result is already visible** in the client (e.g. “subagent result is visible to the user; do not reiterate”), **do not repeat or summarize that output** unless they ask, or unless synthesizing **multiple** subagent results requires it. When they ask for only a **brief third-person subagent-completion line**, provide that single line without replaying subagent contents.
- For **TypeScript `interface` and `type` definitions**, do not attach comments to individual members; place documentation in a **JSDoc block above** the `interface`/`type` (summarize props there, e.g. with `@property` or prose for fields that need explanation).
- Product UI direction favors **softer corners** (rounded rectangles, not near-pill defaults): tune via global `--radius` in `src/app/globals.css` and shared primitives; reserve `rounded-full` for intentional pills (e.g. avatars, true pills), not as the accidental default for buttons and cards.
- When implementing from an attached PRD plan or a read-only reference under `.cursor/plans/`, do not edit that plan file unless the user explicitly asks; track work under **`.agents/plans/<task-folder>/`** (e.g. `todos.md`), plan frontmatter todos, or existing todo items instead. When the user revises a plan or asks to align after an audit, match their scope: if they want only todos updated (including before implementation), sync todos/frontmatter or the relevant **`.agents/plans/...`** todo file—do not edit the plan body unless they ask to.
- **Writable** repo plans and task artifacts live under **`.agents/plans/`**. **`.cursor/plans/`** stays Cursor-native / read-only for attached exports. Do not create new task research, plan, or todo documents under `prds/` unless the user explicitly asks for that legacy location.
- For `.daily-recap/` updates, preserve the existing per-date folder layout (`.daily-recap/YYYY-MM-DD/`) and split recap entries by responsibility/problem solved rather than collapsing multiple days or topics into one folder.
- When a plan specifies a commit order or split (e.g. migration vs tests), match that order in git commits.
- When finishing an **`.agents/plans/...`** slice or explaining why `todos.md` boxes stay unchecked, make intent explicit: tick completed items; for anything left open, note inline or in the review section whether it is optional/deferred, environment-blocked (e.g. full-repo lint resource limits), or post-merge—so unchecked rows do not read as accidental omissions.
- If the user says not to push yet, commit locally only and do not push or assume remote updates.
- When asked to run tests, prefer scoped commands (e.g. `pnpm test:unit:changed`, `pnpm test:integration:changed`, or a single spec file); do not run the full integration or full suite if the user asks to limit scope.
- In Vitest, `vi.mock` / `importActual` must use the **same module specifier string** the production file uses (e.g. `@/hooks/...`). A different path (including a relative path to the same file) can register as a **separate** module id, so the mock may not intercept the dependency under test.
- Before changing code for **CodeRabbit**, review-bot, or similar external findings: **do not treat vendor severity as authoritative**; verify each item against the current tree. Fix **real defects** (bugs, security, wrong behavior) and **MVP-readiness** gaps; skip changes that mostly **clutter** the codebase without clarity or safety wins. Keep fixes **accurate and minimal**.
- When the working tree mixes changes from multiple agents or tasks, commit only files that belong to the current workstream (inspect `git status` / `git diff` before staging).
- For substantial implementation plans, surface multiple viable approaches where tradeoffs differ so the user can choose before coding.
- When asked to clean up or fix **markdown formatting only**, preserve exact wording, paths, and meaning; adjust structure (headings, lists, blank lines, escapes) without paraphrasing or “improving” copy.
- Before closing, reverting, or editing work tied to a **GitHub issue number**, confirm the issue’s actual title and scope (e.g. security vs feature) so actions match the tracked work.
- When running local Playwright smoke on this machine during active development, prefer serial execution over project-level parallelism if the user indicates the machine is resource-constrained; faster on paper is irrelevant if the run becomes less trustworthy.
- When the user asks you to **review or improve a plan/slice markdown file** (e.g. under **`.agents/plans/`**), apply better approaches by **editing existing sections inline**—do not add a trailing “new section” appendage unless they want that structure.
- For duplicate-code or code-health fixes, prefer extracting **tiny semantic helpers** that name shared invariants or reusable UI chrome; avoid generic helpers that hide intent, and leave similar code duplicated when the business meaning may diverge.
- After substantial plan slices or refactors, user **often asks** for a pass over **changed files** to catch **over-abstraction**—prefer **straightforward** shapes when simpler code reads easier.
- When tightening module boundaries, **avoid re-export-only forwarding** for owned implementation modules; **import from definition sites** unless a barrel is an intentional public API surface.
- For Vitest tests that should be skipped when a runtime global is unavailable (for example `DOMException`), prefer declaration-level skips like `it.skipIf(...)` over runtime early returns or `context.skip()` so the runner records the skip explicitly.
- When a handoff-style prompt attaches a plan and says "to-do's have already been created; do not create them again; mark them as in_progress as you work; don't stop until you have completed all the to-dos," honor that contract literally: do not re-create todos, move through the existing list one in_progress item at a time, execute end-to-end without stopping for confirmation, and do not edit the referenced plan file unless explicitly asked.
- For **large multi-step `.agents/plans/` execution**, the user may require **one subagent per plan step** (named model, e.g. composer-2), with that subagent **finishing the step** (including **targeted tests** in the same subagent), **committing** after validation, **not pushing** unless asked, and (when consolidating) **merging the work branch into `develop` locally** without pushing `develop` to remote.
- For token-efficient work, delegate codebase exploration, search, and trivial research to `explore` subagents on a cheaper/faster model (e.g. `composer-2-fast`) and have them report back only the important findings; reserve the parent agent for synthesis and final edits.
- When auditing an **`.agents/plans/<issue>-*`** folder against its GitHub issue, verify listed file paths and responsibility claims against the real tree (for example with an `explore` subagent on a fast model) before changing plan markdown or starting implementation; plans often assume files that are missing, renamed, or already refactored.
- Global skill **`github-issue-to-plan`** lives at `~/.agents/skills/github-issue-to-plan/SKILL.md` (all workspaces on this machine). It loads the issue, finds or **creates** a plan folder named `<issueNumber>-<title-slug>` — for **this repo** prefer **`.agents/plans/`** as the writable location (`AGENTS.md`); then audits gaps—prefer it when the user bridges GitHub issues to local plan artifacts.
- For **“commits from today”** (or similar) code review, check `.daily-recap/YYYY-MM-DD/` for that day’s recap and listed hashes before relying only on `git log --since`.
- When the user asks for **`/review` plus CodeRabbit** (or explicit “code rabbit” alongside a normal review), run both a standard severity-ordered review and a CodeRabbit-oriented pass on the same scope.
- Prefer importing **plan read-projection-layer types** from **`@/features/plans/read-projection/types`** when replacing convenience or barrel-derived paths — that module stays the canonical export surface for shared DTO/UI types from read projection (including **module detail**: `ModuleDetailReadModel`, `ModuleDetailTask`, etc.).
- **Module detail storage vs projection:** `getModuleDetailRows(planId, moduleId, userId)` in [`src/lib/db/queries/modules.ts`](../../src/lib/db/queries/modules.ts) returns adapter rows only; `getModuleDetailForRead` in [`src/features/plans/read-projection/service.ts`](../../src/features/plans/read-projection/service.ts) runs `buildModuleDetailReadModel` from [`src/features/plans/read-projection/module-detail.ts`](../../src/features/plans/read-projection/module-detail.ts) for nav/lock/prev-next and flat task/resource DTOs.

## Learned Workspace Facts

- **App Router `src/app/(group)/` layout:** after moving routes into **route groups** `(marketing)`, `(auth)`, `(app)`, **imports** should use matching aliases — e.g. `@/app/(app)/plans/...`, `@/app/(marketing)/pricing/...`, `@/app/(auth)/auth/...` — so tools and `pnpm check:type` resolve the same tree as `git` paths. If **`pnpm check:type` fails on stale paths** under **`.next/dev/types/`** (e.g. `validator.ts` pointing at pre-move locations), run **`rm -rf .next`** and retry.
- **Product chrome vs marketing:** `SiteHeader` + padded `<main className="flex-1 pt-16">` can live in **per-group** layouts; **marketing** `(marketing)/layout.tsx` may include `SiteFooter`; **auth** and **app** groups typically use header + main **without** the global marketing footer so product routes do not inherit landing chrome.
- **Top navigation breakpoints:** in shared nav, **`lg:` and up** use the desktop header with **visible** primary links; **below `lg`** the layout uses the **mobile** header/sheet pattern — so “tablet” widths can present **hamburger** unless the UI is changed to show links at `md+`.
- **`Pick<Logger, 'error' | 'warn' | 'info'>`** (and similar `Pick` on overloaded methods) **drops overloads**; for injectable loggers (e.g. `model-resolver`), prefer **`{ error: Logger['error']; warn: Logger['warn']; info: Logger['info'] }`** or the full **`Logger`** type so call sites match `@/lib/logging/logger`.
- For **server components and server actions**, prefer `**requestBoundary.component()`** / `**requestBoundary.action()`** whenever the caller needs the current user row or authenticated DB access. The old `**withServerComponentContext()`** / `**withServerActionContext()`** helpers remain compatibility shims; the only verified direct escape hatch is `**getEffectiveAuthUserId()\*\*`for redirect-only identity checks such as`src/app/page.tsx`. The removed `getCurrentUserRecordSafe()` helper is not a supported shortcut for this path.
- Prefer `**@tests/...**` imports for test-only assets (mocks, helpers under `tests/`) when aliases are configured; conventions live in `[tests/AGENTS.md](../../tests/AGENTS.md)`.
- For Vitest specs that need to vary `**NODE_ENV**`, `**VITEST_WORKER_ID**`, or similar env flags, prefer `**vi.stubEnv()**` plus `**vi.unstubAllEnvs()**` over direct `process.env` assignment; the env config layer now treats process env through readonly typing, so direct `NODE_ENV` writes can fail `pnpm check:type`.
- Cross-surface **UI consistency** work should prefer **semantic tokens** in `src/app/globals.css` (`--radius`, `--primary`, `--success`, etc. and `@theme inline` color mappings) and shared components (`Button`, `Badge`, tabs) over ad-hoc Tailwind palette classes (`slate-*`, `green-*`, unrelated gradients) for interactive, completion, and marketing surfaces. For quick audits of a running app, Chrome DevTools MCP against `pnpm dev` (e.g. `navigate_page`, `take_snapshot`, optional `evaluate_script` on buttons/links) helps compare computed radii and colors across routes. Consolidated visual/style reference: `[docs/styles/style-guide.md](../styles/style-guide.md)`.
- **Next.js app network boundary:** route protection and Neon auth middleware run through `[src/proxy.ts](../../src/proxy.ts)` with `PROTECTED_PREFIXES` (not a root `middleware.ts` in this repo). For **true anonymous** manual local `pnpm dev` sessions, shell `env -u DEV_AUTH_USER_ID` (etc.) does not override values loaded from `.env.local`; unset or comment dev-auth-related vars there and restart so anonymous browser sessions are trustworthy. `**LOCAL_PRODUCT_TESTING=false` alone is not enough:** in development, `DEV_AUTH_USER_ID` still makes server code treat requests as that user; clearing true anonymous behavior requires `**DEV_AUTH_USER_ID`unset** (not only toggling`LOCAL_PRODUCT_TESTING`). When running two local `next dev`servers in parallel on different ports, give each one a separate`distDir`(current smoke dirs:`.test-dist/next-smoke-anon`and`.test-dist/next-smoke-auth`) or they will fight over the build lock. `pnpm test:smoke`already handles env layering and dist dirs via`tests/helpers/smoke/mode-config.ts`. For multi-port Playwright smoke, `next.config.ts` `allowedDevOrigins`should list hostnames like`'127.0.0.1'`and`'localhost'`, not full URLs, so `/\_next/\*` assets are allowed during local cross-origin dev requests.
- **Neon Auth middleware vs sign-in URL:** unauthenticated requests to protected routes get `**307**` with `**Location: /auth/sign-in**` (no query). Empty `window.location.search` on the sign-in page after that redirect is **expected**. Neon Auth UI uses a `**redirectTo**` query param in **client-side** flows; app code may use `**redirect_url**` for client-side return paths (e.g. plan detail, streaming errors)—see `[docs/testing/smoke-test-results-2026-04-01.md](../../docs/testing/smoke-test-results-2026-04-01.md)` “Middleware redirect and post-login return.”
- Do not wrap **auth-gated** plan/module page data loaders in React `cache()` keyed only by `planId` / `moduleId`—that can reuse results across different authenticated identities in the same server process. Prefer no memoization for those loaders or a cache key that includes auth.
- Unauthorized redirects to sign-in should use `[ROUTES.AUTH.SIGN_IN](../../src/features/navigation/routes.ts)` and encode the return path (e.g. `redirect_url=${encodeURIComponent(...)}`), consistent with patterns like `[PlanDetailContent.tsx](../../src/app/plans/[id]/components/PlanDetailContent.tsx)` and `[ModuleDetailContent.tsx](../../src/app/plans/[id]/modules/[moduleId]/components/ModuleDetailContent.tsx)`. Several older files still hardcode `/auth/sign-in`; prefer the constant for new code and touch-ups.
- Running `pnpm db:migrate` against Neon can fail with compute-time quota errors; that reflects Neon account/project limits, not necessarily bad migration SQL—use local Postgres for development migrations when Neon is quota-blocked. Local parity with Testcontainers uses shared bootstrap in `tests/helpers/db/bootstrap.ts`; `drizzle.config.ts` resolves `DATABASE_URL_NON_POOLING` → `DATABASE_URL_UNPOOLED` → `DATABASE_URL`—for local dev set all three to the same URL. Local bootstrap scripts must not set `NODE_ENV=test` (use `development` or inherit) so test-only app branches do not run. `tests/helpers/db/truncate.ts` only allows truncating databases whose names match the canonical worker/test shapes (e.g. `*_test`, `*_tests`, `atlaris_test_w<n>`); names like `atlaris_dev` do not match—truncation can be forced with `ALLOW_DB_TRUNCATE=true` only when intentionally overriding that guard.
- PostgreSQL RLS controls which rows a role can see or change; it does not restrict which columns may be updated. Column-level writes use `GRANT` / `REVOKE` on columns (or table-level privileges), not RLS alone.
- Privilege changes that are not modeled in Drizzle schema belong in hand-written SQL migrations; keep the same `REVOKE`/`GRANT` logic in migration, the canonical TS allowlist under `src/lib/db/privileges/` when used, `tests/helpers/db` bootstrap, testcontainers `grantRlsPermissions`, and CI grant steps when they must stay in lockstep.
- Ephemeral Postgres for security/integration tests applies migrations via `pnpm db:migrate` so `pg_policy` and related objects match the migration chain; relying on `drizzle-kit push` alone can drift from migration-defined policy text.
- RLS policy tests (`pnpm test:security` / `vitest --project security`) always run the real suite when invoked; they require Docker (Testcontainers). There is no environment variable to skip that suite. GitHub `ci-pr.yml` and `ci-trunk.yml` include a `security-tests` job; trunk path filters that gate integration-style jobs include `tests/security/**`, `tests/helpers/**`, and `tests/setup/**` so shared test infra changes trigger the right CI runs.
- RLS security tests that assert permission failures should use helpers that account for Drizzle-wrapped Postgres errors (e.g. a shared `expectRlsViolation` that inspects message/cause), not only `rejects.toThrow(/permission denied/)`.
- `src/lib/db/privileges/` is for migration-aligned privilege metadata such as column allowlists; large procedural RLS bootstrap SQL stays under `tests/helpers/db/` (e.g. `rls-bootstrap.ts`), not under `privileges/`.
- Duplication / code-health: prefer **Fallow** (`.agents/skills/fallow-skills`), Oxlint, or consolidation passes for overlap. Example when `fallow` is not on `PATH`: `npm exec --yes --package=fallow -- fallow ...` (same pattern as other ad-hoc tooling in this file).
- For the Vitest `integration` and `e2e` projects, `tests/setup/db.ts` runs `waitForInlineRegenerationDrains()` then `resetDbForIntegrationTestFile()` (and `ensureStripeWebhookEvents()`) in a top-level `beforeEach` (unless `SKIP_DB_TEST_SETUP=true`); when unset, `tests/setup/test-env.ts` defaults `REGENERATION_INLINE_PROCESSING` to `false` so fire-and-forget regeneration drains do not race DB-backed specs unless a file opts in. Per-file hooks that only repeat the same reset add cost without improving isolation. The `unit` and `security` projects do not include this setup file.
- Vitest **integration** project uses `**pool: 'forks'**` with `isolate: true` (Vitest 4 defaults `singleFork: false`), so each test file runs in a fresh forked child process and cross-file in-flight Promises die at the process boundary. **Intra-file** (multiple `it` blocks in the same spec) still share a process and event loop, so the in-flight drain `Set` lock + `waitForInlineRegenerationDrains()` in `tests/setup/db.ts` `beforeEach` is still load-bearing. Do **not** add `poolOptions: { forks: { ... } }` — in Vitest 4 it is no longer a typed per-project field and `tsgo --noEmit` (`pnpm check:type`) rejects it. The historical `pool: 'threads'` mode is what allowed unawaited background promises to leak across files in the same worker; that class of bug stays fixed under forks but the same drain-tracking primitives still cover within-file races.
- Integration Vitest parallelism defaults to **4** workers via `vitest.config.ts` (`INTEGRATION_MAX_WORKERS`, fallback when unset); use a lower value when debugging or on resource-constrained machines. `SKIP_TESTCONTAINERS=true` still forces a single worker (see existing smoke/Testcontainers notes elsewhere in this file).
- Full-repo `pnpm vitest run` in this workspace depends on explicit `sequence.groupOrder` values in `vitest.config.ts`: Vitest 4 aborts project startup when multiple projects with different `maxWorkers` share the default scheduling group.
- For `PUT /api/v1/user/profile`, align `updatedAt` with the **database clock** (for example Drizzle `sql<Date>\`now()`), not` new Date()`in Node, when rows are inserted with Postgres-backed`defaultNow()`—strict millisecond ordering assertions otherwise flake under parallel workers when Node and DB clocks land in the same ms.
- First-run onboarding is easy to underestimate from the README alone: treat `docs/development/environment.md` and `docs/development/local-database.md` as part of the cold path (Neon auth URL and related vars, DB choice). Local `pnpm db:dev:*` helpers assume **macOS Homebrew PostgreSQL 17** in docs; other platforms need an equivalent Postgres or a remote DB URL.
- Drizzle `CHECK` constraints that cap `char_length` should use literal numeric values in `sql` fragments in schema files (avoid interpolating TS constants into `sql` templates for those limits) so generated migration snapshots stay stable; keep caps in `src/lib/db/schema/constants.ts` and validate alignment with drift tests (e.g. title-length specs).
- Tests that require **mock-before-import** order (e.g. `sonner` or other side-effect mocks) should keep the side-effect import first. Prettier does not organize imports, so do not add import-sorting tooling casually.
- The `preferred_ai_model` database enum omits `openrouter/free` even when runtime tier defaults may use that catalog id — persistable settings options and saved-value validation must exclude it and keep tier/runtime fallback separate from “saved preference.”
- OpenRouter streaming usage merge: treat provider cost as authoritative per chunk only when the chunk’s `usage` is an object record (e.g. `isObjectRecord(event.usage)` aligned with `normalizeUsage`); `usage: null` and other non-objects must not clear or retain cost inconsistently with normalization—document in `openrouter-cost-contract.ts` when behavior changes.
- **MVP scope:** PDF upload/extraction/generation has been fully removed for launch; do not reintroduce PDF flows unless the user explicitly revives that surface.
- **PDF removal deploy window:** `docs/development/deploy.md` requires **deploy app that no longer depends on PDF artifacts before** migration `0027` (and kin) that removes `origin='pdf'`. If a new binary runs while the DB still has legacy `origin='pdf'` rows, retry/generation paths that forward `plan.origin` into SSE can emit `plan_start` with `origin: 'pdf'` while the wire schema only allows `ai|manual|template`, so clients may drop the event—normalize legacy origins in emission (or keep parser tolerant) until migration is fully applied fleet-wide.
- When auditing plan-level refactors for missing consumers, check ALL API routes under the relevant path (not just the ones already listed in the plan). The status API route at `[src/app/api/v1/plans/[planId]/status/route.ts](../../src/app/api/v1/plans/[planId]/status/route.ts)` is a read consumer that flows through `getPlanGenerationStatusSnapshot` → `buildPlanDetailStatusSnapshot` → `derivePlanReadStatus`—easy to overlook because it reads like a mutation-adjacent polling endpoint.
- When a plan says "delete a mapper/layer," audit what real semantics it encodes first. The plan read **projection** package lives under `[src/features/plans/read-projection/](../../src/features/plans/read-projection/)` (e.g. `[detail-aggregate.ts](../../src/features/plans/read-projection/detail-aggregate.ts)` for nested sort, `[detail-status.ts](../../src/features/plans/read-projection/detail-status.ts)` for `buildPlanDetailStatusSnapshot` + classification, `[detail-dto.ts](../../src/features/plans/read-projection/detail-dto.ts)` for normalization and `logger.warn`). Each behavior needs a named destination before deletion.
- Status derivation for plans exists at two layers: `[read-status.ts](../../src/features/plans/read-projection/read-status.ts)` exports `derivePlanReadStatus` (generation lifecycle from DB fields); canonical list/summary status uses `deriveCanonicalPlanSummaryStatus` in `[summary-status.ts](../../src/features/plans/read-projection/summary-status.ts)`. Import that canonical helper directly from its definition file; do not re-export it through `[summary-projection.ts](../../src/features/plans/read-projection/summary-projection.ts)`. List/dashboard **display** status (including 30+ day inactivity → `paused`) is centralized in `[selectors.ts](../../src/features/plans/read-projection/selectors.ts)` as `derivePlanSummaryDisplayStatus` (delegates canonical, then applies staleness); `[plan-utils.ts](../../src/app/(app)/plans/components/plan-utils.ts)` and `[activity-utils.ts](../../src/app/(app)/dashboard/components/activity-utils.ts)` call that selector. Do not conflate the generation, summary, and display layers—they use different inputs. Keep retry-exhaustion decisions in the read-projection seam so status route, detail, summary, list, and dashboard consumers reuse one lifecycle truth instead of duplicating active-generation checks.
- Testcontainers Node library auto-spawns a **Ryuk reaper container** (`testcontainers/ryuk:<version>`) on first `start()` in addition to the user's container — Ryuk shows as a second container in OrbStack/Docker Desktop (Testcontainers Desktop hides it) and its mapped port is the library's TCP control channel for label-based reaping if the test process dies unexpectedly. Only one Ryuk runs per host across sessions (the library reuses an existing one). Do not delete or repurpose that second container.
- `pnpm check:type` runs `**tsgo --noEmit`**; the binary ships with `**@typescript/native-preview\*\*`, which must stay in `devDependencies`or CI/pre-push will fail with`sh: 1: tsgo: not found`. The pre-push hook also runs `pnpm check:type`, so type errors anywhere on the working tree (including from missing UI deps unrelated to the diff) will block `git push` — install the missing package or fix the type error rather than bypassing the hook.
- `STRIPE_LOCAL_MODE=true` is rejected by the production-mode guard in `[src/lib/config/env/shared.ts](../../src/lib/config/env/shared.ts)` whenever `NODE_ENV === 'production'`. Because Next.js's `next build` sets `NODE_ENV=production`, CI Build steps cannot use it as an escape hatch — pass the real Stripe price ID secrets (`STRIPE_STARTER_MONTHLY_PRICE_ID`, etc.) through the workflow `env:` block on the Build job instead. The same guard pattern applies to other dev-only env toggles in `shared.ts`.
- Keep `better-auth` and `@better-auth/passkey` pinned to the `**~1.4.x**` range (`>=1.4.9` closes the upstream CVE without bumping minors). Bumping to `1.6.x` breaks the transitive ABI: `@neondatabase/auth-ui` was compiled against 1.4.x typing for `refreshToken` (now non-nullable in 1.6.x) and `@daveyplate/better-auth-tanstack` expects the older `useActiveMember` function shape. Enforce in **both** the direct dep pins and a `pnpm.overrides` entry. For the CVE-bumped `vite ^7.3.2`, pin it as a **direct top-level devDep** so the override actually re-resolves the nested `vitest → vite` chain (pnpm overrides alone do not always force re-resolution of transitive peer-dep matches).
- CI integration jobs (`[.github/workflows/ci-pr.yml](../../.github/workflows/ci-pr.yml)` `integration-light` and `[ci-trunk.yml](../../.github/workflows/ci-trunk.yml)` `integration-tests`) reuse the local Testcontainers + worker-isolated DB stack — no `services: postgres` block and no `pnpm db:push` plumbing. Worker count is controlled by `**INTEGRATION_MAX_WORKERS`\*\* (default 4) plus a `workflow_dispatch` `integration_workers` input (`4`/`2`) so the suite can be downgraded without code changes; trunk shards 2× × 4 workers = up to 8 parallel test files. With `dorny/paths-filter@v3`, `workflow_dispatch` events have no diff to compare so all filters return `true` and manual dispatches still run integration. The CI security-audit gate uses `pnpm audit --prod --audit-level=high` — only prod-affecting high+critical advisories fail the job.
- Test DB safety guards (production-aware): `resetServiceRoleClientForTests` in `[src/lib/db/service-role.ts](../../src/lib/db/service-role.ts)` throws when `NODE_ENV === 'production'`; `dropDatabaseIfExistsWithClient` in `[tests/setup/db-provisioning.ts](../../tests/setup/db-provisioning.ts)` refuses any DB name that does not start with `atlaris_test_`; `assertSafeToTruncate` in `[tests/helpers/db/truncate.ts](../../tests/helpers/db/truncate.ts)` is covered with negative cases in `[tests/unit/helpers/truncate-safety.spec.ts](../../tests/unit/helpers/truncate-safety.spec.ts)`. The internal `dropDatabaseIfExistsWithClient` is the only drop helper that should remain — earlier `dropWorkerDatabaseIfExists` / `withAdminConnection` / public `dropDatabaseIfExists` were removed as dead code.
- Testcontainers integration runs must not share one fixed runtime-state file across separate `pnpm vitest run` processes. `tests/setup/testcontainers.ts` now sets a per-process `TESTCONTAINERS_ENV_FILE`, and `tests/setup/test-env.ts` / `db-provisioning.ts` read that env-scoped path so concurrent local Vitest sessions do not overwrite each other's container metadata and send workers to the wrong `atlaris_test_w<n>` database.
- Inline-drain helpers in `[src/features/jobs/regeneration-worker.ts](../../src/features/jobs/regeneration-worker.ts)` (`isInlineDrainFree`, `registerInlineDrain`, `waitForInlineRegenerationDrains`) are covered by `[tests/unit/jobs/regeneration-worker-inline-drain.spec.ts](../../tests/unit/jobs/regeneration-worker-inline-drain.spec.ts)`.
